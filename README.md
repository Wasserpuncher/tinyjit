# tinyjit

**A JIT compiler in pure Python that emits real x86-64 machine code and runs it on the CPU.**

No `gcc`. No LLVM. No dependencies. tinyjit parses a small C-like language,
compiles each function to actual x86-64 instructions, writes those bytes into
executable memory with `mmap`, and calls them through `ctypes`. From that point
on your CPU is running the code directly — Python is no longer in the loop.

```console
$ python -m tinyjit run examples/fib.tj -e fib 30
832040

$ python -m tinyjit dump examples/fib.tj
; fn fib(1 arg)  offset=0 size=224
  00000000  55 48 89 e5 48 81 ec 20 00 00 00 48 89 bd f8 ff
  00000010  ff ff 48 8b 85 f8 ff ff ff 48 89 85 f0 ff ff ff
  ...
```

Those bytes are not a bytecode for some interpreter to chew on. `55` is
`push rbp`. `48 89 e5` is `mov rbp, rsp`. The CPU decodes and executes them.

## The one number that matters

On this machine (Python 3.14, x86-64), computing `fib(30)`:

| | Time | |
| --- | ---: | --- |
| Native code from tinyjit | **~19 ms** | |
| The included tree-walking interpreter | **~25 000 ms** | ~1000× slower |

That gap *is* the point of a JIT: the same program, the same result, but one of
them is machine code. (Both computed `832040`. The multiplier varies run to run
and with `n`; a summation loop over two million iterations shows a similar
~1000×, while cheap functions like `ackermann(2,3)` show less, because the fixed
cost of a native call dominates.)

## What it can compile

The language is small on purpose — 64-bit signed integers are the only type —
but it is a real language, not a calculator:

```rust
// examples/mathlib.tj
fn gcd(a, b) {
    while (b != 0) {
        let t = b;
        b = a % b;
        a = t;
    }
    return a;
}

fn is_prime(n) {
    if (n < 2) { return 0; }
    let d = 2;
    while (d * d <= n) {
        if (n % d == 0) { return 0; }
        d = d + 1;
    }
    return 1;
}
```

It has functions and recursion (including mutual recursion), `if`/`else`,
`while`, local variables, the arithmetic operators `+ - * / %`, all six
comparisons, and short-circuiting `&&`/`||`/`!`. Up to six parameters per
function, passed in registers per the System V ABI.

## How it works

Four stages, each in its own module and readable on its own:

1. **`syntax.py`** — a hand-written lexer and recursive-descent parser produce an
   AST. Operator precedence is encoded as a ladder of grammar rules.
2. **`interpreter.py`** — a tree-walking evaluator. It is not used to run your
   code (that is the JIT's job); it exists to *define* the language's semantics
   independently, so the compiler can be checked against it.
3. **`compiler.py`** — walks the AST and emits x86-64. The strategy is a **stack
   machine**: every expression leaves its result in `rax`, and to combine two
   values the left operand is spilled to a slot in the function's stack frame
   while the right is computed. Simple enough to get provably right, and it still
   produces native code. Calls between functions are emitted as relocations and
   patched once every function has a known address.
4. **`executor.py`** — asks the OS for memory via `mmap`, copies the code in,
   flips the page to executable with `mprotect`, and builds a `ctypes` function
   pointer. It never leaves a page writable *and* executable at once (W^X).

### The corners that actually take the work

- **64-bit wraparound.** Every operation is defined to wrap in the signed 64-bit
  domain, exactly like the hardware. `fib`, `fact` and friends genuinely
  overflow, and the interpreter overflows the same way, so they still agree.
- **Division that would crash the process.** x86 `idiv` raises a CPU fault on
  divide-by-zero *and* on `INT64_MIN / -1`. Either would kill the Python
  interpreter outright. The compiler branches around both: `/0` and `%0` are
  defined as `0`, and the `-1` case is handled with `neg` so no `idiv` runs. The
  reference interpreter defines identical results, and the tests hammer these
  inputs specifically.
- **Stack alignment.** System V requires the stack 16-byte aligned at each
  `call`. Because intermediate values live in frame slots rather than on the CPU
  stack, `rsp` never moves after the prologue — so alignment is established once
  and every call inherits it.

## How I know it's correct

The generated bytes either compute the right answer or they don't; there is no
room for hand-waving. The suite (`python -m unittest discover -s tests`, 70 tests)
checks this from two directions:

- **Exact encodings.** `test_assembler.py` pins the precise bytes of each
  instruction against the architecture (e.g. `mov rax, rdi` *must* be
  `48 89 f8`), so an encoding regression fails loudly and specifically.
- **Differential execution.** `test_jit.py` compiles dozens of programs, runs
  them on the CPU, and compares every result against the reference interpreter
  over thousands of inputs — every operator crossed with the values most likely
  to break a compiler: `0`, `±1`, sign changes, `2^31`, `2^32`, and the signed
  64-bit extremes `INT64_MIN`/`INT64_MAX`. If one instruction were mis-encoded,
  the answers would diverge and the test would catch it.

That second point is why the interpreter earns its keep: it is the oracle the
JIT is judged against, written in a completely different style so the two are
unlikely to share a bug.

## Install & use

Requires Python 3.10+ on an **x86-64** machine running Linux or Intel macOS.
There is nothing to install:

```console
$ git clone https://github.com/Wasserpuncher/tinyjit && cd tinyjit

# Run a function natively
$ python -m tinyjit run examples/mathlib.tj -e gcd 1071 462
21

# Or drive it from Python
$ python3 -c "import tinyjit; m = tinyjit.compile('fn sq(x){return x*x;}'); print(m.sq(9))"
81

# Disassemble what it generated
$ python -m tinyjit dump examples/mathlib.tj

# Cross-check native output against the interpreter
$ python -m tinyjit run examples/mathlib.tj -e collatz 27 --interp
111
```

## Limitations

These are deliberate; the goal is to understand code generation end to end, not
to compete with a production compiler.

- **x86-64 System V only.** No ARM (so no Apple Silicon natively), no Windows
  ABI, no 32-bit. On an unsupported CPU the JIT refuses with a clear message;
  the interpreter still works everywhere via `--interp`.
- **64-bit signed integers are the only type.** No floats, strings, arrays,
  pointers or structs.
- **Naive code generation.** A stack machine with no register allocator, no
  constant folding, no inlining. The output is real machine code, but an
  optimising compiler would run rings around it.
- **No optimiser and no debugger integration.** `dump` gives you a hex view; it
  does not symbolicate.
- **Single translation unit.** All functions are compiled together into one
  blob; there is no separate compilation or linking against external code.

## Why build this

Because "Python can't do that" is usually a statement about the standard library,
not the machine. The whole pipeline from source text to executing machine code
fits in about 1,000 readable lines, and every layer — parser, code generator,
the `mmap`/`mprotect` dance — is something you can hold in your head. It also
makes concrete a few things that are otherwise abstract: why signed division is
special-cased in every real compiler, what a calling convention actually pins
down, and what a JIT *is* once you strip away the mystique.
